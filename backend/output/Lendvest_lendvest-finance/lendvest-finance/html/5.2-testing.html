This document covers the testing framework and test suite for the LVLidoVault system. It explains how to understand, run, and extend the existing tests, focusing on the comprehensive integration tests that validate the entire epoch lifecycle and risk management scenarios.

For information about setting up the development environment, see Development Environment. For contributing guidelines and code standards, see Contributing.

Test Framework Overview

The LVLidoVault system uses Foundry as its primary testing framework, which provides a fast, flexible testing environment for Ethereum smart contracts. The testing infrastructure is configured to support complex DeFi protocol interactions including flash loans, automated tasks, and multi-party scenarios.

CI Configuration

The CI workflow includes several validation steps:

  • Code Formatting: Ensures consistent code style using forge fmt --check
  • Contract Compilation: Validates that all contracts compile successfully
  • Test Execution: Runs the complete test suite with verbose output
  • Dependency Management: Automatically handles git submodules for dependencies

The workflow is configured to:

  • Run on all pushes and pull requests
  • Use Ubuntu latest as the execution environment
  • Install the nightly version of Foundry for latest features
  • Fail fast on any step failure to provide quick feedback

Testing Best Practices

When working with the LVLidoVault test suite, follow these guidelines:

Mock and Simulation Patterns

  • Price Manipulation: Use stEthPerTokenMock() for simulating market conditions
  • Time Progression: Use vm.warp() to simulate epoch durations and claim periods
  • User Actions: Create distinct addresses for different user types (lenders, borrowers, CLs)
  • Balance Management: Use deal() to provide test accounts with appropriate token balances

Test Isolation

Each test should:

  • Start from the same initialized state (via setUp())
  • Clean up any state changes that might affect other tests
  • Use unique user addresses to avoid interference
  • Reset any mock values to default states

Automation Testing

When testing Chainlink automation:

  • Always check checkUpkeep() before calling performUpkeep()
  • Decode the performData to understand which task will execute
  • Use the forwarder address for automation calls
  • Verify task completion by checking contract state changes
