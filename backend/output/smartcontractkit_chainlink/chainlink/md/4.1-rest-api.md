Overview
--------

The Chainlink REST API uses JSON:API format for requests and responses, with standard HTTP methods (GET, POST, PATCH, DELETE) for resource manipulation. The API supports both session-based authentication for web UI interactions and token-based authentication for programmatic access. All endpoints are versioned, with the current version being v2.

Sources: [core/web/router.go1-103]() [core/cmd/app.go1-349]()

Authentication
--------------

The API supports three authentication methods:

1. **Session-based Authentication**: Used primarily for browser interactions
2. **API Token Authentication**: Used for programmatic access
3. **External Initiator Authentication**: Specialized tokens for external systems that initiate jobs

### Session Authentication

To create a session:

```
POST /sessions
Content-Type: application/json

{
  "email": "user@example.com",
  "password": "your-password"
}
```

A successful login will return a session cookie that must be included in subsequent requests.

### API Token Authentication

API tokens can be created by an authenticated user:

```
POST /v2/user/token
```

This token can then be used by including it in the `Authorization` header:

```
Authorization: Bearer YOUR_API_TOKEN
```

### External Initiator Authentication

External initiators use a pair of headers for authentication:

```
X-Chainlink-EA-AccessKey: YOUR_ACCESS_KEY
X-Chainlink-EA-Secret: YOUR_SECRET
```

Authorization and User Roles
----------------------------

The API implements role-based access control with the following roles:

* **Admin**: Full access to all endpoints
* **Edit**: Can create and modify resources but cannot manage users
* **Run**: Can execute jobs but cannot modify resources
* **View-only**: Can only view resources

Endpoints are protected using role middleware:

```
auth.RequiresAdminRole(handler)  // Only admins can access
auth.RequiresEditRole(handler)   // Admins and editors can access
auth.RequiresRunRole(handler)    // Admins, editors, and runners can access
```

Sources: [core/web/router.go246-269]()

Rate Limiting
-------------

The API implements rate limiting to prevent abuse. Separate limits are configured for:

* Authenticated requests
* Unauthenticated requests
* Static assets
* Index page

Rate limits are defined in the node's configuration.

Pagination
----------

Many endpoints that return lists of resources support pagination using the following query parameters:

* `size`: Number of items per page
* `page`: Page number (starting from 1)

Example request:

```
GET /v2/jobs?page=2&size=10
```

Paginated responses include links to navigate between pages:

Error Handling
--------------

Errors are returned in the JSON:API format:

Common HTTP status codes:

| Status Code | Description |
| --- | --- |
| 200 OK | Request succeeded |
| 201 Created | Resource successfully created |
| 400 Bad Request | Invalid request parameters |
| 401 Unauthorized | Authentication required |
| 403 Forbidden | Insufficient permissions |
| 404 Not Found | Resource not found |
| 422 Unprocessable Entity | Validation error |
| 429 Too Many Requests | Rate limit exceeded |
| 500 Internal Server Error | Server error |
