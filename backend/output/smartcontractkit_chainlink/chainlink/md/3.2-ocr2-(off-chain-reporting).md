OCR2 (Off-Chain Reporting 2) is Chainlink's protocol for decentralized data aggregation and reporting. It enables oracles to aggregate their observations off-chain through a peer-to-peer network before submitting a single transaction with the finalized result to the blockchain. This significantly reduces on-chain gas costs and improves scalability while maintaining security and decentralization.

This document describes the core architecture and components of the OCR2 implementation in the Chainlink node codebase. For information about specific plugin types, see their respective documentation pages.

Architecture Overview
---------------------

OCR2 in Chainlink is designed with modularity at its core, allowing for support of multiple blockchain networks and various data aggregation strategies through plugins. The system is built around a delegate pattern that orchestrates the creation of services specific to each OCR2 job.

Sources: [core/services/ocr2/delegate.go108-128]()), [core/services/relay/evm/evm.go140-175]()

Core Components
---------------

### OCR2 Delegate

The OCR2 Delegate is the central component responsible for creating and managing services for OCR2 jobs. It implements the `job.Delegate` interface and handles job lifecycle events.

The Delegate's main responsibilities include:

1. Creating appropriate services for each OCR2 job type
2. Managing job lifecycle (creation, execution, deletion)
3. Coordinating between relayers, plugins, and the OCR protocol
4. Handling OCR configuration and key management

When a new OCR2 job is created, the Delegate's `ServicesForSpec` method creates the necessary services based on the job's plugin type and relay configuration.

Sources: [core/services/ocr2/delegate.go108-128]()), [core/services/ocr2/delegate.go262-273]()), [core/services/ocr2/delegate.go417-539]())

### OCR2 Database

The OCR2 database layer provides persistent storage for OCR2 job state, configuration, and transmission data. It implements the `ocrtypes.Database` interface from the libocr library.

The database manages several key data types:

1. **Persistent State**: Stores epoch-related data for OCR protocol execution
2. **Contract Config**: Stores on-chain configuration parameters
3. **Pending Transmissions**: Stores reports prepared for on-chain submission
4. **Protocol State**: Stores arbitrary protocol-specific state

The database schema uses tables specific to OCR2, including:

* `ocr2_persistent_states`
* `ocr2_contract_configs`
* `ocr2_pending_transmissions`
* `ocr_protocol_states`

Sources: [core/services/ocr2/database.go18-29]()), [core/services/ocr2/database.go39-66]()), [core/services/ocr2/database.go102-207]()), [core/services/ocr2/database.go209-267]()), [core/services/ocr2/database\_test.go63-102]())

### Relayers

Relayers provide an abstraction layer for interacting with different blockchain networks. Each relayer implements the `commontypes.Relayer` interface and is responsible for creating provider instances for specific chains.

The EVM relayer is the most feature-complete implementation and provides methods for creating various provider types:

1. **PluginProvider**: Standard provider for OCR2 reports
2. **MercuryProvider**: Provider for Mercury price feeds
3. **CCIPProvider**: Provider for Cross-Chain Interoperability Protocol
4. **LLOProvider**: Provider for Low-Latency Oracle feeds
5. **FunctionsProvider**: Provider for Chainlink Functions
6. **OCR3CapabilityProvider**: Provider for OCR3 capabilities

Each provider implements specialized behaviors for its specific plugin type.

Sources: [core/services/relay/evm/evm.go140-175]()), [core/services/relay/evm/evm.go368-400]()), [core/services/relay/evm/evm.go403-453]()), [core/services/relay/evm/evm.go472-546]()), [core/services/relay/evm/evm.go634-668]())

### Plugin System

OCR2 uses a plugin architecture to support different data aggregation and reporting strategies. Each plugin implements specific reporting logic while sharing the common OCR2 protocol infrastructure.

The plugin system is managed through the OCR2 Delegate's `ServicesForSpec` method, which creates the appropriate services based on the plugin type specified in the job. Each plugin has unique configuration requirements and capabilities.

Sources: [core/services/ocr2/delegate.go505-538]()), [core/services/ocr2/validate/validate.go109-136]())

Configuration System
--------------------

OCR2 has a robust configuration system that involves both job-specific configuration and protocol-level parameters.

### Job Configuration

OCR2 jobs are configured through a TOML specification that defines:

1. Plugin type
2. Relay and chain configuration
3. Plugin-specific configuration
4. Bootstrapper peers
5. Transmission/signing keys

This configuration is validated through the `ValidateSpec` function which ensures that the job configuration is complete and coherent.

### OCR Protocol Configuration

The OCR protocol itself requires configuration parameters that control the consensus timing, report generation, and transmission behavior. These parameters include:

* Block confirmation settings
* Timing parameters for rounds and stages
* Transmission schedules
* Maximum data sizes

These parameters are converted from a job specification into a `LocalConfig` that is used to configure the OCR Oracle.

Sources: [core/services/ocr2/validate/validate.go33-77]()), [core/services/ocr2/validate/validate.go95-101]()), [core/services/ocr2/delegate.go162-198]())

OCR2 Job Lifecycle
------------------

The OCR2 job lifecycle is managed by the OCR2 Delegate and involves several stages:

When a job is created, the Delegate's `BeforeJobCreated` method is called, followed by `ServicesForSpec` to create the necessary services. These services are then started by the job manager. When a job is deleted, the `OnDeleteJob` method cleans up resources such as log filters.

Sources: [core/services/ocr2/delegate.go262-273]()), [core/services/ocr2/delegate.go273-414]()), [core/services/ocr2/delegate.go417-539]())

OCR3 Integration
----------------

OCR2 provides integration with OCR3 capabilities through adapter patterns. The system implements adapters for:

1. OCR3 Contract Transmitter
2. OCR3 Onchain Keyring
3. OCR3 Multi-Chain Keyring

These adapters allow OCR2 to use OCR3 capabilities while maintaining backward compatibility. The Multi-Chain Keyring specifically enables OCR3 to work with multiple blockchain networks simultaneously.

Sources: [core/services/ocrcommon/adapters.go24-63]()), [core/services/ocrcommon/adapters.go64-89]()), [core/services/ocrcommon/adapters.go167-260]()), [core/services/ocrcommon/adapters\_test.go106-115]())

Database Schema
---------------

The OCR2 system uses several database tables to store its state:

| Table Name | Purpose | Key Columns |
| --- | --- | --- |
| `ocr2_persistent_states` | Stores protocol state | `ocr2_oracle_spec_id`, `config_digest`, `epoch`, `highest_sent_epoch`, `highest_received_epoch` |
| `ocr2_contract_configs` | Stores contract configuration | `ocr2_oracle_spec_id`, `plugin_id`, `config_digest`, `config_count`, `signers`, `transmitters` |
| `ocr2_pending_transmissions` | Stores reports pending transmission | `ocr2_oracle_spec_id`, `config_digest`, `epoch`, `round`, `report`, `attributed_signatures` |
| `ocr_protocol_states` | Stores arbitrary protocol state | `ocr2_oracle_spec_id`, `config_digest`, `key`, `value` |

The `plugin_id` column in `ocr2_contract_configs` allows for multiple configurations per oracle specification, enabling different plugins to have their own configurations.

Summary
-------

OCR2 (Off-Chain Reporting) is a core component of the Chainlink network that enables efficient, secure, and decentralized data aggregation and reporting. Its modular architecture supports multiple blockchains and reporting strategies through relayers and plugins. The system is designed for flexibility, maintainability, and scalability, allowing for the integration of new capabilities like OCR3 while maintaining backward compatibility.

The OCR2 system is used for a variety of use cases, including:

1. Price feeds (Median, Mercury)
2. Low-Latency Oracles (LLO)
3. Automation (OCR2Keeper)
4. Cross-chain messaging (CCIP)
5. Decentralized computation (Functions)

Each of these use cases builds on the common OCR2 infrastructure while implementing specialized behavior through plugins.
